# Markdownlint Rules Reference

This file explains every rule in `.markdownlint.json`.
If you turn a rule on or off, document the reason here.

---

## How to read this

- `true` — rule is enforced. PRs will fail if they break it.
- `false` — rule is disabled. We consciously chose to allow this.
- `"default": true` — all rules are on by default unless explicitly turned off below.

---

## Rules we have turned off

### MD013 — Line length
**Status: disabled**
Markdown files in this repo contain prompt rules and tables.
Enforcing a line length limit would break table formatting and make profiles harder to read.

### MD033 — Inline HTML
**Status: disabled**
Some profile examples include HTML code blocks for documentation purposes.
Blocking inline HTML would prevent these examples from being shown.

### MD041 — First line should be a top-level heading
**Status: disabled**
Profile files start with comment-style headers like `# Name:` and `# Works with:`.
These are metadata fields, not document headings.
Enforcing H1 as the first line would conflict with the profile header format.

### MD024 — Multiple headings with the same content
**Status: disabled**
Profiles across the repo use the same section names: Output, Rules, Behavior.
These headings repeat by design — each profile is a standalone file.
Enforcing unique headings would force artificial renaming of standard sections.

### MD022 — Headings should be surrounded by blank lines
**Status: disabled**
Some profile sections are intentionally compact.
Requiring blank lines around every heading adds unnecessary vertical space in short profiles.

### MD025 — Single title (H1) per document
**Status: disabled**
Profile headers use multiple `#` comment lines at the top (Name, Works with, Best for, etc).
Markdown parses these as H1 headings, which would trigger this rule incorrectly.

### MD032 — Lists should be surrounded by blank lines
**Status: disabled**
Profile rules use tight lists by design.
Adding blank lines around every list would make profiles visually noisy and harder to copy-paste.

### MD040 — Fenced code blocks should have a language specified
**Status: disabled**
Some code blocks in profiles show generic examples without a specific language.
Requiring a language on every code block would add noise to simple examples.

### MD031 — Fenced code blocks should be surrounded by blank lines
**Status: disabled**
Same reason as MD032 — profiles use compact formatting intentionally.

### MD026 — Trailing punctuation in headings
**Status: disabled**
Some section headings in profiles end with a colon for readability.
For example: `## Per-tool Notes` style conventions should not be penalized.

## Rules that are on (default: true)

Everything not listed above is enforced.
Key rules that are active:

- **MD001** — Heading levels increment by one only (no jumping from H1 to H3)
- **MD003** — Consistent heading style (ATX style: `##` not underline style)
- **MD004** — Consistent unordered list style
- **MD005** — Consistent list indentation
- **MD009** — No trailing spaces
- **MD010** — No hard tabs
- **MD011** — No reversed link syntax
- **MD014** — Dollar signs in shell code blocks should not be used before commands
- **MD018** — Space after hash in headings
- **MD034** — No bare URLs — use proper link syntax
- **MD037** — No spaces inside emphasis markers
- **MD038** — No spaces inside code span markers
- **MD039** — No spaces inside link text
- **MD042** — No empty links
- **MD045** — Images should have alt text
- **MD046** — Consistent code block style
- **MD047** — Files should end with a single newline

## Changing a rule

1. Update `.markdownlint.json`
2. Add or update the entry in this file with the reason
3. Include both files in your PR
